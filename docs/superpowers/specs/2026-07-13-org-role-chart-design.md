# 团队架构「岗位组织图」重设计

> 状态：设计提案，待实现。2026-07-13。
>
> 参考方向：Pingboard 式岗位组织图。保留 Coboard 已有的部门 / 小组 / 岗位、
> 岗位名额、多人任职、兼任和岗位申报能力，不照搬传统“一人一个节点”的员工汇报树。

## 1. 为什么重做

当前「图谱」以星系画布为默认模式，适合探索团队成员，但不适合快速回答组织架构最常见的
问题：

- 团队分成哪些部门和小组；
- 每个岗位负责什么、由谁任职；
- 哪些岗位仍有空缺；
- 一个人在哪些岗位兼任；
- 新成员可以申请什么岗位。

新设计将信息主语从“画布上的视觉节点”改为“组织单元与岗位”。部门 / 小组负责分支，
岗位负责承载人员与名额，成员不再单独撑开树宽。

## 2. 产品原则

1. **先读结构，再看人员**：第一眼看到部门和岗位层级，第二眼看到任职者。
2. **岗位是稳定节点**：人员加入、离开或兼任不会改变整张图的空间结构。
3. **空缺也是信息**：无人任职或未满员的岗位必须留在图中，并能进入申报流程。
4. **兼任不画蜘蛛网**：同一成员可以出现在多个岗位卡中，用“兼任”标记和详情汇总表达，
   不在主画布绘制跨分支连线。
5. **阅读与编辑分离**：普通成员获得干净的组织图；管理员在显式编辑态中看到操作入口。
6. **50 人以内一屏可导航**：允许缩放和平移，但不要求用户理解特殊的空间隐喻。
7. **Monochrome 为底**：界面主体沿用 Coboard 现有黑白灰体系，点缀色只用于识别和状态，
   不使用大面积彩色区块。

## 3. 页面结构

在现有导航中增加第四个顶层视图入口：

- **图谱**：保留现有星系 / 树形组织图及内部模式切换；
- **岗位图**：新的岗位组织图；
- **列表**：保留现有可排序、可移动的树形管理视图；
- **招募**：保留现有岗位申报、审批和“我的申报”。

岗位图不加入原图谱右下角模式切换，而是在页面顶部与图谱、列表、招募并列。现有
`OrgPlanetCanvas`、`OrgChartCanvas` 和 `coboard-org-chart-mode` 偏好完整保留。

页面头部新增搜索框；右下保留缩放、适应视图和 100% 控制。

## 4. 信息架构

### 4.1 合成团队根节点

数据库允许多个根部门，因此画布在渲染时创建一个**只读合成根节点**：

- 标题：团队；
- 内容：去重后的在岗人数；
- 可选：负责人头像（全局管理员不自动等于团队负责人）；
- 不写入 `org_nodes`，不参与编辑和拖拽。

只有一个根节点时也显示团队根，保持所有组织图的入口一致。

### 4.2 部门 / 小组分支头

结构节点使用 monochrome 分支头，与普通岗位卡形成轻微层级差异：

- 左侧：部门或小组名称；
- 右侧：该子树的**去重成员数**；
- 第二行可选：负责人头像与姓名；
- 部门和小组主体均使用中性背景；
- 顶层部门可获得一个稳定的低饱和点缀色，仅用于顶部 2px 色线、小圆点和选中态；
- 小组继承最近祖先部门的点缀色，但默认只显示在类型圆点上；
- 可折叠，折叠后显示“岗位数 · 人数 · 空缺数”。

点缀色只帮助区分分支，不承载唯一语义；类型仍通过文字和图标表达。关闭点缀色后，
组织图的信息层级仍须完整成立。

### 4.3 岗位卡

每个 `kind='position'` 节点只渲染一张岗位卡，即使岗位允许多人任职：

- 标题：岗位名称；
- 右上：在岗 / 名额，例如 `2/3`；名额不限时显示 `2/∞`；
- 任职者：头像 + 姓名，负责人带皇冠；
- 最多直接显示 3 人，更多显示 `+N`，点击展开完整成员；
- `headcount > members.length` 时显示一行虚线空缺槽位：`空缺 1 人`；
- 无人任职时岗位卡仍完整显示，不退化为空白占位；
- 岗位描述在点击后的详情层显示，不常驻画布。

同一成员出现在两个及以上岗位时，在姓名后显示轻量“兼任”标记；悬停或点击可查看其
全部任职路径。

### 4.4 非岗位节点的直属成员

现有模型允许成员直接挂在部门或小组。为兼容已有数据：

- 分支头下方增加一张“直属成员”卡；
- 使用与岗位卡相同的头像 / 姓名布局；
- 不虚构岗位名称，不把直属成员自动迁移到岗位；
- 管理员可从该卡进入成员管理。

长期建议通过产品引导把正式分工沉淀为岗位，但本期不修改数据约束。

## 5. 布局

### 5.1 桌面端

采用“结构分支横向展开、同组岗位纵向堆叠”的布局，接近参考图：

```text
                         ┌────────────────────┐
                         │ 团队          18 人 │
                         └─────────┬──────────┘
                 ┌────────────────┼────────────────┐
       ┌─────────┴────────┐ ┌─────┴────────────┐ ┌─┴────────────────┐
       │ 内容部       6 人 │ │ 产品部       7 人 │ │ 增长部       5 人 │
       ├──────────────────┤ ├──────────────────┤ ├──────────────────┤
       │ 内容负责人   1/1 │ │ 产品负责人   1/1 │ │ 增长负责人   1/1 │
       │ 头像  姓名       │ │ 头像  姓名       │ │ 头像  姓名       │
       ├──────────────────┤ ├──────────────────┤ ├──────────────────┤
       │ 文案编辑     2/3 │ │ 产品设计     2/2 │ │ 渠道运营     1/2 │
       │ 头像 姓名 +1     │ │ 头像 姓名 +1     │ │ 头像 姓名 / 空缺 │
       │ 空缺 1 人        │ ├──────────────────┤ └──────────────────┘
       └──────────────────┘ │ 质量工程     1/1 │
                            └──────────────────┘
```

布局规则：

- 分支列宽 `264px`，同级列间距 `40px`；
- 分支头高 `56px`，岗位卡最小高 `88px`，根据成员行数向下增长；
- 同一结构节点的直属岗位纵向排列，间距 `12px`；
- 子部门 / 子小组作为下一层横向分支，不与直属岗位混成同级卡；
- 父子关系使用 1.5px 浅色圆角正交线；
- 布局结果包含节点矩形、边和整体边界，供自动适应视图使用。

### 5.2 布局算法

新增纯函数 `role-layout.ts`：

```ts
roleLayout(roots): {
  branches: RoleBranchLayout[];
  cards: RoleCardLayout[];
  edges: RoleEdgeLayout[];
  bounds: Rect;
}
```

算法分两步：

1. 后序计算每个结构节点需要的水平跨度；叶结构节点至少占一列；
2. 前序定位：父节点居中于所有结构子分支，直属岗位只增加当前列高度，不增加树宽。

岗位节点按现有 `rank` 排序。结构子节点也按 `rank` 排序，但始终在直属岗位区块之后进入
下一层，避免岗位与部门争夺同一层级。

### 5.3 移动端

宽度小于 `768px` 时不缩小整张画布，而切换为纵向大纲：

- 部门 / 小组为可折叠分组；
- 岗位卡占满容器宽度；
- 保留搜索、岗位空缺和申报入口；
- 不显示连线和画布缩放控件。

## 6. 交互

### 6.1 阅读态

- 点击部门 / 小组头：折叠或展开该分支；
- 点击岗位卡：打开岗位详情抽屉，展示职责、完整成员、名额和组织路径；
- 点击任职者：展示成员摘要及全部兼任岗位；
- 点击空缺槽位：切换到「招募」视图并定位该岗位；
- 搜索支持成员名、岗位名、部门名；命中后展开祖先、居中并短暂高亮；
- Esc 清除搜索定位；`0` 适应视图，`+/-` 缩放。

岗位详情优先复用现有 `OrgMembersDialog` 与 `OrgNodeDialog` 的数据和 mutation，视觉上可改为
Drawer；本期不重复实现成员和岗位编辑表单。

### 6.2 编辑态

只有全局管理员可进入：

- 分支头悬停显示：新增子级、编辑、成员、更多；
- 岗位卡悬停显示：编辑岗位、管理任职者、删除；
- 新增岗位默认挂到当前部门 / 小组末尾；
- 重排和跨层级移动继续在「列表」视图完成，本期不在画布实现拖拽；
- 所有删除继续使用现有级联影响确认。

不在画布提供拖拽是有意选择：组织调整属于低频高影响操作，按钮式移动比自由拖拽更容易
验证父级、排序和作用域约束。

## 7. 岗位空缺与申报联动

空缺数：

```text
headcount === null  → 不显示空缺数
headcount > occupied → vacancy = headcount - occupied
headcount <= occupied → 已满
```

这里的 `occupied` 沿用服务端规则：岗位上的 lead 和 member 都占用名额。

- 普通成员点击空缺岗位后进入招募详情，可申报或撤回；
- 已满岗位不显示申请动作；
- 管理员看到待审批数量徽标，点击进入该岗位的审批区；
- 图谱只展示状态摘要，完整申请表和审批仍留在「招募」视图。

## 8. 数据与接口

本期**不需要数据库迁移**。直接复用：

- `org_nodes.kind / title / description / headcount / parent_id / rank`；
- `org_node_members.role / user_id / rank`；
- `org_applications` 及现有申请、撤回、录用、拒绝接口；
- `GET /org/tree` 返回的节点和成员；
- 现有 SSE `org` 失效事件。

前端新增派生索引：

- `uniqueMembersBySubtree`：分支人数按 userId 去重；
- `positionsByUser`：判断兼任并生成完整任职路径；
- `vacanciesByPosition`：岗位空缺和满员状态；
- `searchIndex`：结构标题、岗位标题、成员姓名的归一化搜索文本。

所有派生逻辑写成纯函数并单测，不散落在卡片组件中。

## 9. 视觉规格

### 9.1 Monochrome 基底

- 画布背景：`bg-background` 或极弱的 `bg-muted/20`，不使用星点、轨道、渐变和辉光；
- 卡片：`bg-card border-border rounded-xl`，默认无阴影，悬停使用轻微 `shadow-sm`；
- 分支头：`bg-muted/50`，与岗位卡依靠字重、间距和边框区分，不靠大色块区分；
- 连线：统一使用 `border` token，1.5px，位于卡片下层且不穿过卡片；
- 正文、次级文字、禁用态严格沿用 `foreground / muted-foreground`；
- 不写死白、黑或具体灰阶，深色模式完全由现有 token 驱动。

### 9.2 点缀色

允许使用点缀色，但限制在单个组件面积的约 10% 以内：

- 顶层部门分支头顶部 2px 色线；
- 部门 / 小组名称前的 6px 类型圆点；
- 当前搜索命中或键盘聚焦时的 ring；
- 待审批数量徽标；
- 空缺岗位的申请动作 hover / focus；
- 负责人皇冠等已有业务标记。

点缀色从低饱和 `indigo / rose / teal / amber / violet / sky` 调色板中稳定分配，但不得用于
整块分支头背景、整张岗位卡背景或大面积连线。普通状态徽标保持 monochrome。

### 9.3 卡片细节

- 岗位名称为主要文字，姓名为次级文字；
- 头像 32px，卡片内最多三行成员；
- 空缺槽使用中性虚线边框和 muted 文案，不使用警告红；
- 满员状态使用中性色，只有待审批状态允许点缀色；
- 兼任使用中性描边徽标，不再引入额外颜色；
- hover 以边框加深、轻微上移或阴影表达，避免整卡换色。

## 10. 无障碍

- 所有卡片和分支头均可通过 Tab 聚焦；
- 展开状态使用 `aria-expanded`；
- 图标按钮必须有 `aria-label` 和 Tooltip；
- 人数、空缺和兼任都有文字，不能只通过颜色或图标表达；
- 缩放不改变浏览器页面缩放；
- `prefers-reduced-motion` 下关闭居中和展开动画；
- 移动端大纲保持与桌面画布相同的 DOM 阅读顺序。

## 13. 验收标准

1. 用户无需缩放即可辨认顶层部门和主要岗位；
2. 10 秒内能找到某人所属部门、岗位及兼任关系；
3. 所有有限名额岗位都能直接看出在岗数和空缺数；
4. 50 人、30 个岗位的组织图无卡片重叠或连线穿卡；
5. 图谱、列表、招募使用同一份组织数据，更新后无状态不一致；
6. 手机端可完整浏览和操作，不依赖横向拖动画布；
7. 岗位图作为独立入口，原图谱的星系 / 树形切换和偏好保持不变。

- 不从任务协作关系自动推导组织架构；
- 不在画布实现自由拖拽重组；
- 不改变现有岗位申报权限规则。
